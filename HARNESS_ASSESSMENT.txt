  ---
  BrokeLLM — Agentic Harness Evaluation

  Classification: code agent + embedded AI feature
  Harness shape: Single-process gateway orchestrator with policy-driven model
  routing and event-harness evaluation
  Architecture level: ~Level 2-3 (durable harness + extensible surface, but
  approaching complexity limits for solo maintenance)

  ---
  FINDINGS

  F1 — Tool/Execution Boundary: Shim PATH Trust Model

  Severity: HIGH — safety gap
  bin/broke:293-314 creates a shims/ directory and prepends it to $PATH. Every
  entry is a symlink to _harness_shim.py, but nothing validates that symlinks
  haven't been replaced between creation and execution. A compromised agent
  could swap a symlink to point at an attacker binary.
  Why it matters: The entire safety story of harness mediation depends on all
  tool calls flowing through _harness_shim.py. A swapped symlink bypasses the
  event ledger entirely.
  Operator impact: Silent execution — no event is logged for hijacked commands.
  Fix direction: After prepare_harness_shims(), create an immutable manifest of
  expected symlink targets (e.g., hash each link destination). At execution
  time, validate before exec. Better: use execve() with the absolute binary path
   instead of relying on $PATH resolution, or bwrap --bind-mount each shim
  individually.

  F2 — Missing Permission Tiers for Harness Commands

  Severity: HIGH — missing primitive
  The shim classifies commands into verification:test, verification:task, vcs,
  runtime, tool — but no policy engine blocks or gates anything based on
  classification. The cwd_allowed() check is the sole permission gate. A git 
  push --force or rm -rf classified as vcs executes without approval.
  Why it matters: The skill's permission tier model (always-allow / ask-first /
  never-allow) is absent.
  Operator impact: Workers can execute high-risk actions through the shim
  without review.
  Fix direction: Add a policy.deny / policy.require_approval table keyed by
  command + argument patterns. At minimum, deny destructive verbs (rm, dd, mkfs,
   git reset --hard) regardless of cwd scope.

  F3 — Workflow State Conflated with Logging

  Severity: MEDIUM — durability risk
  The harness writes events to events.jsonl with chained hashes (good), but
  there is no explicit "waiting", "retry", "approved", or "blocked" state
  machine. The harness knows only run.registered → command.started →
  command.completed → verdict.issued. If a worker crashes mid-task, there's no
  retry_scheduled or compensated state.
  Why it matters: Crashes leave runs in implicit "running" state forever; no
  resumability path.
  Operator impact: Stuck runs require manual cleanup; no visibility into what
  was in flight.
  Fix direction: Add a status field to the run record that progresses through:
  registered → executing → collecting_checkpoint → awaiting_verdict → 
  completed/failed/escalated. Write state changes atomically. Add a broke 
  harness recover <run-id> command.

  F4 — Event Hash Chain Not Validated on Read

  Severity: MEDIUM — audit integrity gap
  Events are chained with prev_event_hash, but nothing verifies the chain on
  _read_json() or at verdict time. A malicious process could modify an earlier
  event without detection unless someone manually replays the hashes.
  Why it matters: Chain integrity is a write-time-only guarantee; tampering is
  detectable only if explicitly checked.
  Fix direction: Add broke harness verify-chain <run-id> that walks the file and
   confirms each event_hash matches recomputation. Consider validating on
  verdict issuance.

  F5 — Evaluation Coverage Is Thin for Production-Harness Claims

  Severity: MEDIUM — missing primitive
  33 tests, all in test_mapping.py. Zero tests for:
  - _proxy.py request handler error paths (e.g., 502 from LiteLLM, malformed
  JSON body)
  - _harness_shim.py policy enforcement (e.g., cwd violation, classification
  accuracy)
  - _socket_bridge.py connection behavior (idle timeout, slot exhaustion)
  - Import/export integrity (overwrite flag, malformed JSON)
  - Profile inheritance over team policies
  By the skill's rule: "do not evaluate from vibes alone" — the harness claims
  balanced/high_assurance verdicts but has no tests confirming those verdicts
  produce correct outcomes under adversarial input.
  Operator impact: Regression risk on every edit to proxy policy or harness
  verdict logic.
  Fix direction: Minimum add: proxy handler unit tests (mock upstream), shim
  cwd/classification tests, verdict correctness matrix with known
  inputs/outputs.

  F6 — Context Assembly Has No Budget Discipline

  Severity: LOW — maintainability
  cmd_config() generates ALL models for ALL key variants into a single
  config.json LiteLLM file. There's no concept of "load only the tools relevant
  to the current context." If a user has 2 keys per provider across 6 providers
  plus fallback chains, the config file sprawls.
  Operator impact: Larger config → slower provider startup → higher chance of
  config errors.
  Fix direction: Dynamic profile generation — emit only active models + fallback
   chain for the current team/slot mapping.

  F7 — No Cost/Token Visibility

  Severity: LOW — operational gap
  cmd_metrics parses prometheus success/failure counts from LiteLLM, but there's
   no token or cost tracking, no cost-per-lane visibility, and no "runaway
  behavior indicator." The entire value proposition (route to free backends)
  lacks the single metric that proves it's working: cost saved.
  Operator impact: Cannot validate ROI or detect a backend drifting from free to
   paid tier.
  Fix direction: Parse litellm_total_tokens and litellm_response_tokens from
  metrics; derive per-request cost estimates based on provider pricing tables.

  F8 — Duplication Across Components

  Severity: LOW — maintainability degradation
  The following logic is duplicated across files:
  - sha256_text / sha256_obj / canonical_json — replicated in _mapping.py,
  _harness_shim.py, and _proxy.py
  - File locking (locked_file / _locked_file) — two independent implementations
  - read_json with fallback / read with parsing — three versions

  This is exactly the pattern solo-devs regret at 6 months.
  Fix direction: Create bin/_harness_common.py (or reuse _mapping.py as the
  single source of truth) for shared hashing, JSON, locking, and event append.
  Import everywhere.
